# UI Audit Report - Downstream Processing Simulator

Audit date: 2026-05-11  
Repository state inspected: `main` at `296e35e` / release `v0.8.9`  
Audit perspective: computational-simulation scientist and scientific-instrument UI designer  
Primary scope: Streamlit UI under `src/dpsim/visualization`

## Executive Assessment

The Downstream Processing Simulator UI is scientifically coherent and unusually strong for a research-grade simulator. Its main design choice, a lifecycle workflow from target definition through M1 fabrication, M2 functionalisation, M3 column operation, run execution, validation, and calibration, matches the mental model of downstream process development rather than a generic analytics dashboard. The simulator also makes a serious attempt to prevent false precision through evidence-tier banners, decision-grade metric rendering, and calibration-aware result suppression.

The current UI is reliable enough for exploratory process-development simulation, screening studies, calibration planning, pressure-envelope checks, and wet-lab SOP preparation. It is not designed, and should not be presented, as a validated production instrument-control interface, GMP batch-record system, or regulatory release tool.

The main limitations are not in the scientific intent. They are in information architecture, panel density, responsive layout risk, and maintainability of large stage modules. Experienced downstream-processing users will recognize the scientific workflow, but they may still have to work too hard to determine which result is current, which panel owns a decision, and how an optimizer recommendation maps back to bench-actionable physical settings.

Overall audit ratings:

| Area | Rating | Summary |
|---|---:|---|
| Scientific workflow fit | High | The lifecycle structure maps well to real DSP development. |
| Model/UI integration | Medium-high | Recipe state, calibration state, and evidence tiers are connected thoughtfully. |
| Decision-grade result honesty | High | Tier-gated rendering is a major strength. |
| Panel layout and operator ergonomics | Medium | Functionally rich, but dense and sometimes hard to scan. |

## Intended Role of the Current Responding Model

I found no evidence that the application currently integrates an LLM-style conversational responder inside the UI. The active "responding model" from the user's standpoint is the scientific simulation stack selected and parameterized through the UI: the global Scientific Mode, the lifecycle orchestrator, calibration store, recipe state, pressure envelope tools, column-operation calculations, uncertainty propagation, and decision-grade render rules.

The UI presents three scientific modes:

- `Empirical`: fastest and most screening-oriented.
- `Hybrid`: the default balance of calibration, mechanistic behavior, and robustness.
- `Mechanistic`: stricter, more physics-first, and more sensitive to missing calibration.

This model is intended to respond to a structured `ProcessRecipe` rather than free-form user intent. The UI controls define target, M1, M2, and M3 settings; the lifecycle runner then executes or stages the simulation, stores the result, and renders tier-qualified outputs. Calibration data and evidence tiers affect whether results are shown as precise numbers, intervals, rank bands, or suppressed values.

This is the correct model role for a scientific process-development simulator. It keeps the user in control of physical assumptions and avoids presenting the simulator as an autonomous decision maker.

The main risk is semantic opacity. A user can select Empirical, Hybrid, or Mechanistic globally, but the UI should more explicitly state at run time what that selection changes:

- Which equations or solver pathways are active.
- Which calibration requirements are enforced or relaxed.
- Which outputs may be suppressed or downgraded.
- Whether runtime, uncertainty, or pressure-envelope behavior will change.

The current model integration is structurally sound, but the operator-facing explanation of mode consequences should be strengthened near the Run stage and result panels.

## Scientific Workflow Evaluation

The lifecycle-first structure is the UI's strongest scientific design choice. The stages are:

1. Target
2. M1 matrix fabrication
3. M2 functionalisation
4. M3 column operation
5. Run
6. Validation
7. Calibration

This sequence reflects how a downstream scientist would reason about an adsorbent and column operation: first define product and operating envelope, then define matrix and chemistry, then column transport and operation, then validate and calibrate.

The design is better aligned with real process development than a flat dashboard because it preserves dependency order. M3 operation depends on M2 chemistry; M2 depends on M1 substrate properties; validation depends on all upstream assumptions. The UI also exposes this dependency explicitly through stage status, chain warnings, and tier rollups.

Strengths:

- The pipeline spine gives users a process-development map.
- Direction A and Direction B views support both sequential workflow and triptych inspection.
- M3 run controls are guarded by upstream M2 availability.
- Validation makes evidence capping visible across M1, M2, and M3.
- Calibration is visible as a first-class lifecycle concern, not hidden in settings.
- Wet-lab ingestion and SOP export connect simulation outputs back to bench execution.

Weaknesses:

- The Stage 07 Calibration area now contains several distinct mental models: forward uncertainty, inverse posterior fitting, multi-column design, wet-lab ingestion, and optimization access. These are scientifically related, but not all are calibration. This risks confusing the user's sense of where a task belongs.
- Optimization output is still more simulation-internal than bench-actionable. Normalized design coordinates are useful for implementation and debugging, but a DSP scientist needs physical settings, constraints, uncertainty, and pressure/safety feasibility as the primary output.
- The M3 panel combines setup, safety checks, run execution, detector traces, SOP export, and run comparison. This is powerful but dense.

Judgment:

The workflow is scientifically valid and supports the intended simulation functions. The next improvement should be information architecture refinement, not a change in scientific direction.

## Structural Reliability

The UI has several reliability mechanisms that are appropriate for a scientific simulator.

### Strengths

Single recipe state:

- `ui_recipe.py` stores a serialized `ProcessRecipe` in session state.
- UI controls sync into the recipe through explicit helpers.
- This reduces stale in-memory dataclass identity problems across Streamlit reruns.

Lifecycle result storage:

- `ui_workflow.py` stores lifecycle output in `lifecycle_result`.
- Legacy aliases are maintained for older M2/M3 views.
- This preserves compatibility while the lifecycle architecture evolves.

Run context integration:

- The lifecycle runner passes the calibration store, mobile phase, isotherm spec, geometry, and flow conditions into the execution path.
- Background execution includes cancellation polling.

Autowiring:

- `shell/autowire.py` detects new lifecycle results, captures snapshots, appends run history, caches breakthrough data, and derives stage statuses.

Decision-grade rendering:

- `decision_grade_render.py` centralizes whether metrics appear as exact numbers, intervals, rank bands, or suppressed outputs.
- This is essential for avoiding overstated precision.

Regression checks:

- UI test coverage is broad for a Streamlit application.
- The 272 passing tests include visualization, recipe state, UI contracts, chrome smoke checks, and dossier-related behavior.

### Reliability Risks

Large UI modules:

| File | Approximate size | Risk |
|---|---:|---|
| `tabs/tab_m1.py` | 1585 lines | Hard to audit and refactor safely. |
| `tabs/tab_m3.py` | 1369 lines | High-density stage with many state transitions. |
| `tabs/tab_m2.py` | 826 lines | Moderate complexity and layout risk. |
| `panels/isotherm_selector.py` | 25 KB | Specialized panel with substantial embedded logic. |
| `shell/triptych.py` | 22 KB | Important layout surface with custom behavior. |

These files are not automatically wrong, but they raise the cost of maintaining panel consistency, adding viewport tests, and preventing state regressions.

Streamlit module reload behavior:

- `app.py` force-reloads selected core modules on import.
- This improves interactive Streamlit development, but it can create subtle object-identity and performance risks.
- The current tests appear to account for enum-value comparisons and reload behavior, but this should remain a watched area.

Custom HTML/CSS coupling:

- The UI uses `st.html`, `unsafe_allow_html`, custom CSS tokens, Streamlit `data-testid` selectors, and layout selectors such as `:has(...)`.
- This gives the app a polished instrument-style shell, but it increases sensitivity to Streamlit DOM changes.

Responsive layout:

- The right run rail uses a fixed rail width around 380 px.
- The top bar uses multiple fixed columns and dense controls.
- Without browser screenshot testing, narrow-display reliability remains unproven.

Judgment:

The structural design is reliable enough for research simulation workflows, especially because state and evidence semantics are tested. The main structural risk is continued growth of large stage files and custom layout CSS without screenshot-based regression coverage.

## Panel and Layout Evaluation

### Top App Bar

The top bar communicates identity, mode, evidence tier, view direction, and theme. This is appropriate for a scientific instrument-style simulator because the operator should always know the model fidelity and evidence context.

Strengths:

- Scientific Mode is globally visible.
- Evidence rollup is visible in the chrome.
- Direction A/B switching supports both pipeline and workbench workflows.
- The app avoids hiding critical scientific status in a settings drawer.

Risks:

- The top bar is crowded.
- Fixed-width column composition may wrap poorly on narrow displays.
- The difference between scientific mode, evidence tier, and validation stage may not be immediately clear to new users.

Recommendation:

Add a compact run-preflight explanation that states: selected mode, expected model pathway, missing calibration consequences, and outputs likely to be downgraded.

### Pipeline Spine

The seven-stage pipeline is a good structural metaphor. It maps well to downstream process development and supports dependency reasoning.

Strengths:

- Clear lifecycle ordering.
- Good fit for M1 -> M2 -> M3 scientific dependencies.
- Query-param navigation makes stages addressable.

Risks:

- Stage 07 has become a container for too many distinct functions.
- The optimizer is accessed through calibration even though inverse design is not the same as calibration.

Recommendation:

Consider separating:

- Calibration
- Uncertainty
- Inverse design / optimization
- Series design
- SOP / export

This does not require abandoning the lifecycle. It clarifies user intent.

### Run Rail

The run rail is one of the strongest operator-facing elements. It surfaces last-run KPIs, breakthrough preview, evidence rollup, pending edits, and run controls.

Strengths:

- Persistent right-side status matches instrument-console expectations.
- KPIs such as DBC10, recovery, HETP, and Delta P are appropriate for chromatography users.
- Pending edits and baseline comparison help prevent accidental interpretation of stale runs.
- Stop/run controls are close to result context.

Risks:

- The fixed-width rail may consume too much space on laptops or tablets.
- The rail's information density could compete with the active stage panel.

Recommendation:

Implement viewport-specific rail modes:

- Full rail on wide desktop.
- Collapsed KPI rail on medium screens.
- Bottom drawer or modal detail on narrow screens.

### M1 Panel

The M1 panel appears scientifically rich and supports matrix/substrate setup. It also carries substantial complexity.

Strengths:

- M1 is treated as a real lifecycle dependency rather than a hidden input table.
- The panel supports fabrication-specific assumptions and physical descriptors.

Risks:

- The module size indicates a dense panel with many responsibilities.
- A user may struggle to identify the minimum inputs required for a credible M1 run.

Recommendation:

Split M1 into smaller view components:

- Required physical descriptors.
- Fabrication controls.
- Derived properties.
- Evidence and calibration notes.
- Advanced settings.

### M2 Panel

The M2 panel appropriately represents functionalisation as the chemistry bridge between matrix and column operation.

Strengths:

- M2 sits in the correct dependency position between M1 and M3.
- The UI supports chemistry-specific status and downstream handoff.

Risks:

- The relation between M2 result quality and M3 decision quality should remain visually explicit.
- Users need immediate visibility into which M2 assumptions cap M3 confidence.

Recommendation:

Expose "M3-impacting M2 assumptions" in a compact summary near the M3 entry point.

### M3 Panel

The M3 panel is the most important downstream-processing UI surface because it corresponds most directly to chromatography operation.

Strengths:

- Column geometry and flow controls are explicit.
- Derived geometry values are shown.
- M3 run is disabled until upstream M2 data exists.
- Pressure-envelope and Q-dP feasibility checks are present.
- Results include breakthrough, detector traces, pressure-flow context, SOP export, and comparison tools.

Risks:

- Setup, safety, execution, and interpretation are all in one large panel.
- A scientist may need to scroll extensively before reaching the decision result.
- It may be unclear whether a displayed result came from the current recipe, a previous lifecycle result, or a direct M3 run.

Recommendation:

Restructure M3 into a stable operator flow:

1. Column and method setup.
2. Feasibility and pressure envelope.
3. Run controls.
4. Decision summary.
5. Diagnostic traces.
6. SOP and comparison.

The content mostly exists. The improvement is visual ordering and ownership.

### Validation Panel

The validation panel provides a scientifically important evidence ladder and capping logic.

Strengths:

- Evidence capping across M1, M2, and M3 is visible.
- Legacy detailed views are kept in an expander, which reduces default complexity.
- Validation is presented as a lifecycle stage rather than an afterthought.

Risks:

- Validation can appear as a result-review page rather than an actionable confidence-improvement guide.

Recommendation:

For each weak tier, show the next best calibration action and expected impact on rendered outputs.

### Calibration and Inverse Inference

Calibration support is strong for a simulator. The UI includes forward Monte Carlo, inverse posterior fitting, multi-column series tooling, and wet-lab ingestion.

Strengths:

- Calibration is visible and operational.
- The inverse inference tool blocks underdetermined fitting.
- Wet-lab ingestion closes the loop from experiment to model.
- The UI recognizes that calibration changes decision quality.

Risks:

- The inverse inference default table has 3 rows while the fit requires at least 8 observations. This is scientifically correct as a blocker but weak as an onboarding default.
- Multi-column series design appears under Calibration despite being explicitly described as design-time, not calibration.
- Calibration, uncertainty, and inverse design are conceptually overloaded in one stage.

Recommendation:

Seed inverse inference with an 8-row template or provide an "add required template" action. Move multi-column design and optimization toward a dedicated inverse-design area.

### Optimization Panel

The optimization tab is scientifically useful but not yet fully bench-actionable.

Strengths:

- It distinguishes "best predicted" from "best actionable".
- It uses decision-grade metric rendering.
- It recognizes evidence tiers and feasibility.

Risks:

- Normalized 7-D coordinates are not the primary representation a downstream scientist needs.
- Users need physical recipes: pH, conductivity, residence time, gradient slope, column size, loading, flow rate, buffer conditions, and pressure limits.
- Optimizer output should be integrated with SOP export and feasibility gating.

Recommendation:

Make the first optimizer output a physical recipe card with uncertainty, pressure feasibility, evidence tier, and expected tradeoffs. Keep normalized coordinates in an advanced/debug section.

## Evidence-Tier and Decision-Grade Rendering

The evidence-tier system is a major scientific and UI strength.

Observed strengths:

- A global tier banner is mounted in the app shell.
- Decision-grade rendering centralizes metric display rules.
- Weak evidence can produce intervals, rank bands, or suppressed values.
- The UI avoids bare exact metrics when confidence is insufficient.
- Plot and metric annotations connect scientific confidence to visual output.

This is exactly the right pattern for simulation software used by scientists. It reduces the chance that users will treat uncalibrated or weakly calibrated simulations as validated measurements.

Remaining improvements:

- Explain "why this metric is suppressed" directly at the result location.
- Show the minimum calibration action needed to promote the metric.
- Provide per-output confidence traces so users know whether uncertainty comes from M1, M2, M3, or sparse wet-lab data.

## Experienced DSP User Perspective

From the standpoint of a user experienced with chromatography and downstream process development, the UI has a credible scientific shape. The following capabilities are especially relevant:

- Target-to-column lifecycle mapping.
- Column geometry and flow controls.
- Breakthrough and detector-trace outputs.
- DBC10, recovery, HETP, and pressure-related KPIs.
- Pressure-envelope preflight.
- Wet-lab SOP export.
- Calibration and inverse fitting.
- Evidence-qualified decision rendering.

Likely points of friction for this user:

- Too many advanced controls are visible in dense panels.
- The difference between lifecycle run, direct stage run, cached result, and baseline comparison needs constant clarity.
- Optimizer results must be translated into bench settings before they are operational.
- Calibration and inverse design should feel like distinct workflows.
- On smaller screens, a fixed run rail plus dense stage controls could make the app feel cramped.

The application behaves more like a research console than a commercial instrument HMI. That is acceptable for the current product positioning, but the UI should keep distinguishing:

- simulated result
- calibrated prediction
- weak-evidence ranking
- pressure-feasible method
- bench-ready SOP

## Reliability of the Existing Panel Design

The existing panel design is reliable in the sense that it reflects valid scientific dependencies and is backed by tests. It is less reliable as a long-term operator interface because several panels carry too many responsibilities.

Reliable aspects:

- The top-level process sequence is correct.
- The recipe-state pattern is sound.
- Run outputs are evidence-qualified.
- Calibration affects confidence and display behavior.
- The run rail gives persistent context.
- Regression tests protect many UI contracts.

Less reliable aspects:

- Layout stability across screen sizes is uncertain.
- Panel complexity makes it harder to guarantee that the most important decision output is always visible.
- Some workflow concepts are colocated for implementation convenience rather than user intent.
- Large modules increase the chance that future changes will create inconsistent panel behavior.

Conclusion:

The panel design supports the intended workflows, but it should be considered a strong research UI rather than a finished scientific-instrument HMI. It is adequate for expert users who understand the process. It would benefit from more operator-guided hierarchy before being used by less technical process engineers or in formal method-transfer contexts.

## Recommendations

### Priority 0 - Protect Current Reliability

1. Add browser-based visual regression tests.
   - Desktop wide viewport.
   - Laptop viewport.
   - Narrow viewport.
   - Dark and light themes.
   - Direction A and Direction B.
   - At least Target, M3, Run, Validation, and Calibration stages.

2. Add explicit result-source and stale-output labeling.
   - Lifecycle result.
   - Direct M3 result.
   - Cached baseline.
   - Previous run history.
   - Current recipe pending edits.

3. Pin and test Streamlit upgrades.
   - The UI depends on Streamlit DOM behavior and custom CSS.
   - Treat framework upgrades as UI regression events.

### Priority 1 - Improve Scientific Usability

1. Make Scientific Mode consequences visible in Run.
   - Show active model pathway.
   - Show calibration requirements.
   - Show outputs likely to be downgraded.
   - Show runtime and uncertainty implications.

2. Convert optimizer results into physical recipe cards.
   - Include pH, conductivity, residence time, gradient, load, flow, column geometry, and pressure feasibility.
   - Keep normalized coordinates in an advanced section.

3. Restructure M3 around the operator decision flow.
   - Setup.
   - Feasibility.
   - Run.
   - Decision summary.
   - Diagnostics.
   - SOP/export.

4. Seed inverse inference with a valid minimum template.
   - Default to at least 8 measurement rows.
   - Provide import validation and immediate sufficiency status.

### Priority 2 - Reduce Maintenance Risk

1. Break large tab modules into smaller components.
   - Each component should own one section and receive a typed view model.
   - Keep stage orchestration thin.

2. Add layout contracts for reusable components.
   - KPI cards.
   - Tier badges.
   - Pressure envelope summaries.
   - Run-control sections.
   - Result cards.

3. Consolidate calibration, inverse design, and series design terminology.
   - Use stage names that map to scientist intent.
   - Avoid placing design-time tools under Calibration unless they are explicitly calibration tasks.

### Priority 3 - Improve Instrument-UI Polish

1. Add keyboard and accessibility review.
2. Provide local font fallback for locked-down lab machines.
3. Review small text sizes in dense scientific panels.
4. Ensure icon-only controls have stable accessible labels and tooltips.
5. Add print/export views that preserve evidence tier and simulation disclaimers.

## Final Judgment

The Downstream Processing Simulator UI is scientifically credible and directionally well designed. Its strongest qualities are lifecycle alignment, calibration awareness, evidence-tier honesty, pressure-feasibility thinking, and persistent run context. These qualities are exactly what a downstream-processing simulator needs.

The UI's main risk is that its scientific richness is concentrated into dense panels and large modules. Experienced users will understand the domain, but they still need clearer separation between setup, feasibility, run execution, validation, calibration, inverse design, and SOP export. The optimizer in particular should move from normalized model coordinates toward physical, bench-actionable recipes.

The current design is suitable for research-grade simulation and process-development screening. With stronger viewport testing, clearer model-mode explanations, better result-source labeling, and a refined information architecture, it can become a more reliable scientific-instrument-style interface for expert users.
