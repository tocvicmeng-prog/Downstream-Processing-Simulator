# DPSim v0.6.3 — Joint Architecture, Science, and Orchestration Audit

**Auditors:** `/scientific-advisor` + `/architect` + `/dev-orchestrator` (joint)
**Subject:** `Downstream-Processing-Simulator` v0.6.3 (commit `eea8776`)
**Scope:** functional / operational logic errors, intent ↔ as-built conflicts, scientifically infeasible elements, missing invariants
**Date:** 2026-05-04
**Mode:** read-only audit; no code edits

---

## 1. System understanding (joint)

DPSim is a three-module screening simulator. The intended dataflow is `ProcessRecipe → M1 → M2 → M3 → ProcessDossier`, with the calibration store providing tier-promotion feedback. Each module's solver is documented in `CLAUDE.md` / `README.md` and is supposed to obey:

- M1 PBE solver: LSODA (non-stiff moments). Cahn-Hilliard for cellulose NIPS: spectral semi-implicit.
- L2 / L3 / L4: family-specific solvers, mostly BDF for stiff network kinetics.
- M2 reaction kinetics: Radau (chosen for the cyanuric staged kinetics + EDC/NHS).
- M3 LRM: BDF default, BDF for gradient elution, LSODA for catalysis packed bed.
- Zonal PBE (CFD-PBE coupling): LSODA via `solve_ivp`.
- Evidence-tier inheritance: enforced in `RunReport.compute_min_tier`; AST gate (`tests/test_v9_3_enum_comparison_enforcement.py`) protects against Streamlit reload aliasing.

The intent is internally coherent. The defects below are not architectural overhauls — they are precision gaps, doc drift, and small invariant breaks that erode trust.

## 2. Findings (graded BLOCKER / MAJOR / MINOR)

### BLOCKERS

None found that prevent correct operation of any current code path. The serious-looking candidates (zonal-PBE bit-exactness, dual M1 codepath) turned out to be documentation overclaims (see MAJOR-1 and MAJOR-3).

### MAJOR

#### MAJOR-1 — "Bit-exact 0.000000 % rel error" claim is mathematically impossible and contradicted by the test

- **Module:** `src/dpsim/cfd/zonal_pbe.py` (claim) ↔ `tests/test_cfd_zonal_pbe.py:344-395` (reality).
- **Evidence:** `CHANGELOG.md` v0.6.2 entry: "Bit-exact reduction to bare PBESolver in the 1-zone degenerate case (**0.000000 % rel error**)." The README and the new repo README both repeat this. The actual test asserts `pytest.approx(d32_legacy, rel=1e-9)` (1 part per billion) — not byte-identical. The volume-balance test even admits "modulo LSODA tolerance, ≪ 1 %" (`tests/test_cfd_zonal_pbe.py:418`).
- **Mechanism:** `solve_ivp` with `method='LSODA'` is a variable-step adaptive integrator. Two separate calls — one inside `integrate_pbe_with_zones`, one inside the bare `PBESolver` — can take internally different step sequences even with identical RHS, ICs, and tolerances. The output is provably *not* byte-identical; it is bounded by the solver tolerances. Rel-error ≤ 1e-9 is a strong claim; "0.000000 %" is unfalsifiable.
- **Recommended fix:** downgrade the claim everywhere to "agreement at integrator-tolerance level (rel ≤ 1e-9 on aggregated d₃₂)" — the test's actual contract. Replace the `0.000000 %` text in `CHANGELOG.md`, `README.md`, and the `zonal_pbe.py` module docstring. This is a one-line edit per file but matters for scientific integrity (and IP claims).
- **Severity rationale:** MAJOR because the false-precision claim could be exposed in code review, journal review, or due-diligence for the IP audit — and the reviewer would be right.

#### MAJOR-2 — Two parallel taxonomies for "M2 step kind" with no mapping table

- **Modules:** `src/dpsim/core/process_recipe.py:20-40` (`ProcessStepKind`) ↔ `src/dpsim/module2_functionalization/modification_steps.py:77-92` (`ModificationStepType`).
- **Evidence:** `ProcessStepKind` enumerates {`ACTIVATE`, `INSERT_SPACER`, `ARM_ACTIVATE` (v0.5.2), `COUPLE_LIGAND`, `METAL_CHARGE`, `QUENCH`, …}. `ModificationStepType` enumerates {`ACS_CONVERSION`, `PROTEIN_COUPLING`, `QUENCHING`, …}. The vocabularies overlap conceptually but use different identifiers (`ACTIVATE` vs `ACS_CONVERSION`; `QUENCH` vs `QUENCHING`; `COUPLE_LIGAND` vs `PROTEIN_COUPLING`).
- **Mechanism:** Recipes use `ProcessStepKind` (recipe-level vocabulary). Implementation dispatches on `ModificationStepType` (kinetic-solver vocabulary). The translation between them is a hidden-coupling point — any new step kind must be added in *both* taxonomies in lock-step, with the mapping logic kept in sync. The CHANGELOG v0.5.2 entry mentions "PROTEIN_COUPLING dispatch allowlists expanded for new ACS-conversion / arm-activation reagents," confirming the allowlist is a known surface.
- **Recommended fix:** introduce an explicit `STEP_KIND_TO_MODIFICATION_TYPE` mapping table (one source of truth) and a regression test that asserts every `ProcessStepKind` value resolves to either a `ModificationStepType` or an explicit `None` (for non-modification steps like `WASH`, `EQUILIBRATE`). This shuts the door on the v0.5.2-class incident where new step kinds weren't covered everywhere.
- **Severity rationale:** MAJOR because the v0.5.2 patch already proves the gap is real and was caught only by external review.

#### MAJOR-3 — README claims `pipeline/` hosts "legacy L1-L4 kernels"; it doesn't

- **Module:** `README.md` repo-structure block ↔ `src/dpsim/pipeline/`.
- **Evidence:** `pipeline/` contains `__init__.py`, `batch_variability.py`, `checkpoint.py`, `logger.py`, `orchestrator.py` — **no L1-L4 solver kernels**. The actual L1-L4 kernels live in `level1_emulsification/`, `level2_gelation/`, `level3_crosslinking/`, `level4_mechanical/`. `pipeline/` is an orchestration shell with batch-variability, checkpointing, and logging — not legacy kernels.
- **Mechanism:** The misleading description (which I introduced in the README rewrite based on the prior README's wording, see **AUDIT NOTE A** below) primes a new contributor to look in the wrong directory for L1-L4 logic. It also hides the actual purpose of `pipeline/` (orchestration support) behind a "legacy" framing.
- **Recommended fix:** update the README's `pipeline/` line to reflect reality — orchestration support (batch variability, checkpointing, logging). Drop the "legacy L1-L4 kernels" framing.
- **Severity rationale:** MAJOR because new contributors will waste real time chasing a phantom duplication, and because this is a one-edit fix.

#### MAJOR-4 — `_phase_rank` integer ranks contradict the comments above them

- **Module:** `src/dpsim/core/recipe_validation.py:365-376`.
- **Evidence:** The block-comment immediately above the dict says: "v0.5.2 (codex P2-1 fix): `ARM_ACTIVATE` has phase rank **2.5** (after `INSERT_SPACER`, before `COUPLE_LIGAND`)." The dict actually maps:

  ```python
  ACTIVATE: 1, INSERT_SPACER: 2, ARM_ACTIVATE: 3,
  COUPLE_LIGAND: 4, METAL_CHARGE: 5
  ```

  — the integer is **3**, not **2.5**. The G6 ordering produced is functionally correct (monotonic with `ARM_ACTIVATE` between `INSERT_SPACER` and `COUPLE_LIGAND`), but the comment is a lie. A future reader trying to reconstruct the v0.5.2 fix will be confused.
- **Recommended fix:** either (a) change the comment to "rank 3 (after rank 2 INSERT_SPACER, before rank 4 COUPLE_LIGAND)", or (b) renumber to (1, 2, 2.5, 3, 4) using floats and keep the comment. (a) is simpler.
- **Severity rationale:** MAJOR because the recipe-validation FSM is the canonical M2-correctness enforcer; doc drift here erodes the audit trail for the v0.5.2 codex fix.

#### MAJOR-5 — G6 error message omits `ARM_ACTIVATE` from the "required order" string

- **Module:** `src/dpsim/core/recipe_validation.py:393-397`.
- **Evidence:** When G6.1 fires `FP_G6_SEQUENCE_OUT_OF_ORDER`, the error message reads:

  > Required order: ACTIVATE → INSERT_SPACER → COUPLE_LIGAND → METAL_CHARGE.

  The canonical post-v0.5.2 order is `ACTIVATE → INSERT_SPACER → ARM_ACTIVATE → COUPLE_LIGAND → METAL_CHARGE`. A user who hits this error and follows the message will be told their recipe is wrong without being told `ARM_ACTIVATE` exists, then they re-submit and hit the same error from a different root cause.
- **Recommended fix:** update the error string to include `ARM_ACTIVATE` between `INSERT_SPACER` and `COUPLE_LIGAND`.
- **Severity rationale:** MAJOR because user-facing error messages are the primary recovery surface.

### MINOR

#### MINOR-1 — AST gate function name and docstring claim broader scope than the implementation

- **Module:** `tests/test_v9_3_enum_comparison_enforcement.py:131-153`.
- **Evidence:** The test function is named `test_no_identity_or_bare_equality_comparisons_against_enforced_enums` and its docstring says it fails on `<expr> is <Enum>.<MEMBER>` *or* `<expr> == <Enum>.<MEMBER>` patterns "without `.value` on the other side". The actual `visit_Compare` (line 98) only flags `ast.Is, ast.IsNot` — `==` is *deliberately* excluded (see the class-docstring rationale at lines 16-20). The function docstring contradicts the class docstring and the implementation.
- **Recommended fix:** rename to `test_no_identity_comparisons_against_enforced_enums`; rewrite the docstring to match what the AST checker actually does (only `is` / `is not`).
- **Severity rationale:** MINOR — the test is functionally correct; the gap is purely cosmetic / documentation.

#### MINOR-2 — `EXEMPT_FILES` carries a comment about a phantom second exemption

#### MINOR-3 — `cad/README.md` "datatypes.py mismatches" section is stale post-v0.6.0

#### MINOR-4 — CFD validation rule numbering is non-contiguous

- **Module:** `src/dpsim/cfd/zonal_pbe.py:160-202` (`CFDZonesPayload._cross_field_consistency`).
- **Evidence:** The validator references "Rule 2", "Rule 3", and "Rule 6" but no Rule 1, 4, or 5 inline. The rules likely live in `cad/cfd/zones_schema.md`. Without inline numbering, a future reader cannot tell whether rules 1/4/5 are enforced elsewhere or were dropped.
- **Recommended fix:** either inline the full rule list in a docstring at the top of `CFDZonesPayload`, or replace the numeric references with descriptive labels (e.g. "unique zone names", "exchange endpoints", "volume-weighted ε aggregation").
- **Severity rationale:** MINOR — readability.

#### MINOR-5 — Streamlit `use_container_width` deprecation warnings

- **Module:** `src/dpsim/visualization/` (multiple files).
- **Evidence:** The running Streamlit session log shows repeated warnings: "Please replace `use_container_width` with `width`. `use_container_width` will be removed after 2025-12-31." Today is 2026-05-04 — the deprecation has *already passed*. The app currently still works because Streamlit hasn't enforced the removal, but it is on borrowed time.
- **Recommended fix:** migrate all `use_container_width=True` → `width='stretch'` and `use_container_width=False` → `width='content'`. Estimate: 30-60 occurrences based on log frequency.
- **Severity rationale:** MINOR until Streamlit actually removes the kwarg — at which point it becomes a BLOCKER. Tracking now is cheap.

#### MINOR-6 — `st.components.v1.html` deprecated in favour of `st.iframe`

- **Module:** `src/dpsim/visualization/components/impeller_xsec_v3.py:95`.
- **Evidence:** Same Streamlit log: "Please replace `st.components.v1.html` with `st.iframe`. `st.components.v1.html` will be removed after 2026-06-01." That deadline is **27 days away** (today is 2026-05-04).
- **Recommended fix:** migrate the live cross-section component (`impeller_xsec_v3.py:95`) and any other `components.html` callsite to `st.iframe` before 2026-06-01.
- **Severity rationale:** MINOR-trending-MAJOR as the date approaches.

#### MINOR-7 — `evidence_tier` literals in 30 files

- **Module:** multiple. 98 occurrences of `ModelEvidenceTier.{VALIDATED|CALIBRATED|SEMI|QUALITATIVE|UNSUPPORTED}` across 30 source files. Concentrations:
  - `visualization/shell/triptych.py`: 14
  - `visualization/shell/stage_panels.py`: 14
  - `optimization/objectives.py`: 7
  - `module3_performance/catalysis/packed_bed.py`: 4
  - `lifecycle/orchestrator.py`: 3
- **Evidence:** Many uses are legitimate (display strings, manifest construction, value comparisons). But the concentration in `optimization/objectives.py` (7 hits) and `catalysis/packed_bed.py` (4 hits) suggests possible hardcoded defaults that should derive from upstream `ModelManifest.evidence_tier`. Worth a focused review.
- **Recommended fix:** one-shot grep + read of the 7 hits in `optimization/objectives.py` and the 4 in `catalysis/packed_bed.py` to confirm whether each literal is (a) a comparison reference, (b) a final-output tier that should derive from inputs, or (c) a calibration-store sentinel. If (b), refactor to derive.
- **Severity rationale:** MINOR pending the focused review; could escalate to MAJOR if a derived-tier should be flowing but is being short-circuited.
